# Scriptorium 0.4.0 — release notes

**Released:** 2026-04-28

v0.4 ships **discipline enforcement**. The three disciplines that have always
defined Scriptorium — evidence-first claims, the PRISMA audit trail, and
contradiction surfacing — are now enforced at the runtime, the skill, and the
artifact layer in a way that is objectively testable in both **Claude Code**
and **Cowork**.

This release lands as two coupled layers:

- **Layer A — discipline enforcement.** Shared injection, phase-state
  tracking, HARD-GATEs in phase-critical skills, a Scriptorium-specific
  verification skill, and a backwards-compatible advisory rollout.
- **Layer B — reviewer and extraction enforcement.** Extraction isolation
  (Claude Code parallel subagents; Cowork backend dispatch over `mcp` /
  `notebooklm` / `sequential`), a synthesis-exit reviewer gate, and an
  audited override path that blocks publish until explicitly bypassed.

## What's new

### Layer A — session injection and phase-state

- `skills/using-scriptorium/INJECTION.md` is the canonical discipline
  preamble. **Claude Code** injects it via a `SessionStart` hook; **Cowork**
  exposes the same prose through the `scriptorium-mcp` server's MCP
  `instructions` field. One source of truth, two delivery mechanisms.
- `phase-state.json` is the new authoritative artifact for "where in the
  review are we, and what gate decisions stand." It records the current
  phase, every gate's signature over upstream inputs, and any reviewer or
  override decisions. Mutating an upstream artifact invalidates the
  signature and **auto-downgrades** the phase so a stale gate cannot
  silently outlive its inputs.
- `using-scriptorium` opens with an explicit **runtime probe**, then
  dispatches to the right skill. A defensive skill-router fallback fires
  if the probe doesn't match any phase, so an off-script user request
  still lands somewhere sane.

### Layer A — HARD-GATEs and verification

- Phase-critical skills (`lit-screening`, `lit-extracting`,
  `lit-synthesizing`, `lit-publishing`) now open with a HARD-GATE block. A
  runtime that ignores the injected discipline preamble still hits a
  stop-the-world gate inside the skill body.
- New skill `verification-before-completion-scriptorium` enforces a
  Scriptorium-specific completion check (cite scan against `evidence.jsonl`,
  audit-row presence for the just-completed phase, contradictions file
  exists when synthesis claims to be done) before any phase can be declared
  finished. The Claude Code `PostToolUse` hook on `scriptorium verify`
  remains as a redundant defense-in-depth.
- Each phase skill now carries a **red-flag table** and **runtime-honesty
  wording** so the user sees the degradation contract before the skill
  starts running, not after.

### Layer B — extraction isolation

- **Claude Code:** extraction is orchestrated as parallel subagents,
  bounded by the new `extraction_parallel_cap` config key (default 3,
  validated 1–8). The dispatcher refuses to silently exceed the cap.
- **Cowork:** extraction dispatches over one of three backends, decided by
  the runtime probe at session start:
  - `mcp` — `scriptorium-mcp` reachable; orchestrator calls
    `mcp__scriptorium__extract_paper(review_dir, paper_id)` per paper.
    Highest isolation grade.
  - `notebooklm` — NotebookLM enabled, no scriptorium-mcp; orchestrator
    creates a fresh notebook per paper (or rotates a single scratch
    notebook under quota pressure). High isolation when notebooks are
    fresh; medium when rotating.
  - `sequential` — degraded path. Single chat thread, papers one at a
    time, batch-of-five checkpoints, `parallel_cap` ignored. Audit row is
    explicit about the degradation; we do **not** claim parity with
    `mcp` / `notebooklm`.

The chosen literal lands on every `extraction.dispatch` audit row, so an
auditor can confirm the runtime did what the probe announced.

### Layer B — reviewer gate and audited override

- The synthesis-exit reviewer gate runs two reviewers — `cite` and
  `contradiction` — over `synthesis.md` and `evidence.jsonl` before
  publish.
  - **Claude Code:** the two reviewers run as parallel subagents.
  - **Cowork:** the orchestrator picks `notebooklm` (fresh-context branch)
    or `inline_degraded` (drafting-context only, warning row, no parity
    claim) based on the probe.
- A failed gate **blocks publish**. Override is explicit, signature-bound,
  and append-only — and is invalidated automatically if any upstream
  artifact changes after the override is recorded. The override flow is
  exposed via the `scriptorium phase override` CLI and the
  `mcp__scriptorium__phase_override` MCP tool.
- The `scriptorium.mcp` server exposes six tools to the Cowork side:
  `extract_paper`, `finalize_synthesis_reviewers`, `phase_show`,
  `phase_set`, `phase_override`, `verify_gate`.

## Migration and advisory rollout

v0.4 is **advisory by default**. Existing reviews keep working without any
change.

1. **Bump the package**: `pipx upgrade scriptorium-cli` (or
   `pipx install scriptorium-cli==0.4.0`).
2. **Backfill phase state**: for each existing review, run
   `scriptorium migrate-review --to 0.4 <review-dir>`. The command is
   idempotent and fails closed on corrupted state. It writes the new
   `phase-state.json` artifact, computes signatures over the existing
   upstream files, and appends a `migrate_review` row to `audit.jsonl`.
3. **Stay advisory**: by default `enforce_v04 = false` in
   `~/.scriptorium/config.toml`. The new gates surface warnings but do not
   block.
4. **Flip to blocking** when ready: set `enforce_v04 = true` in
   `~/.scriptorium/config.toml`. From then on, HARD-GATE blocks, the
   reviewer gate, and the override invariant are enforced.

If a review predates v0.4 and you do **not** want to migrate it, leave it
alone — `enforce_v04 = false` keeps it usable in legacy advisory mode.

## How v0.4 preserves the three disciplines

- **Evidence-first claims.** The synthesis-exit cite reviewer and the
  Claude Code `PostToolUse` `scriptorium verify` hook both check that
  every claim in `synthesis.md` resolves to a row in `evidence.jsonl`.
  HARD-GATE blocks in `lit-synthesizing` and `lit-publishing` refuse to
  proceed if this invariant is violated.
- **PRISMA audit trail.** Every gate decision, reviewer outcome, override
  event, extraction dispatch, and runtime-degradation event appends to
  `audit.jsonl`. `phase-state.json` carries the signature graph; the
  human-readable `audit.md` is regenerated from `audit.jsonl` so the
  trail is reconstructable end-to-end.
- **Contradiction surfacing.** The contradiction reviewer at the
  synthesis-exit gate refuses to pass a synthesis whose evidence groups
  hide explicit disagreement. `lit-contradiction-check` retains its
  group-by-concept structure and now contributes audit rows the gate can
  consult.

## Cowork smoke matrix

`docs/cowork-smoke.md` is the runtime-degradation contract. v0.4 adds two
new sections you should re-run on every release:

- **Extraction backend matrix** — names the expected per-paper flow for
  `ISOLATION_BACKEND ∈ {mcp, notebooklm, sequential}` and the audit-row
  literal each backend stamps.
- **Reviewer branch matrix** — names the expected per-gate flow for
  `REVIEWER_BRANCH ∈ {notebooklm, inline_degraded}` and the four
  audit-row quartet a Cowork gate run must produce.

Run the matrix whenever `using-scriptorium`'s probe changes, whenever a
phase-critical skill changes, or before tagging a release.

## Compatibility notes

- v0.3 reviews remain readable. The plain `lit-review` flow continues to
  work; new gates surface warnings until you flip `enforce_v04 = true`.
- The exit-code table gains `E_REVIEWER_INVALID = 22` for reviewer-validation
  failures. All other v0.3 exit codes are preserved.
- The PyPI distribution name remains `scriptorium-cli`. The console
  command, import path, and plugin name remain `scriptorium`.

## Deferred after v0.4

The following are explicitly **not** in v0.4 and remain follow-on work:

- Removing the `enforce_v04` flag entirely (will land once advisory rollout
  has soaked).
- Closing any honest runtime degradation that remains after the Cowork
  branches ship — `inline_degraded` and `sequential` are still degraded
  paths.
- Additional reviewer types beyond cite and contradiction.
- Any UX or documentation polish not required for execution or release
  safety.
